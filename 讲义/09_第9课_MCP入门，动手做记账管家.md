**前几节课，我们让 AI 会聊天、会按模板“一键出活”，但它始终是“嘴上功夫”——只会说，不会做。**这节课的 MCP，就是给 AI 装上“手脚”：让它能真正去记一笔账、查一次数据、调用一个工具。这是从“会聊天的 AI”迈向“能办事的 AI”的关键一步。

## 9\.1　先搞懂：MCP 到底解决什么问题

MCP 的全称是 **Model Context Protocol（模型上下文协议）**，由 Anthropic 在 2024 年底提出，如今已经成了行业通用标准，Claude、Cursor、VS Code 等主流工具都支持它。

它解决的痛点是这样的：在没有统一标准之前，你想让 AI 用上某个工具（比如查数据库、读你的文件、发邮件），就得为“每一个 AI”和“每一个工具”单独写一套对接代码。3 个 AI × 5 个工具，就要写 15 套，这叫 **N×M 问题**，谁都受不了。

MCP 的思路就像 **USB\-C 接口**：以前充电器、U 盘、耳机各有各的插头，乱成一团；有了 USB\-C 这个统一标准，一个口什么都能插。MCP 就是 AI 和外部工具之间的那个“统一插口”——工具按 MCP 标准实现一次，所有支持 MCP 的 AI 都能用。

|**对比**|**没有 MCP**|**有了 MCP**|
|---|---|---|
|对接方式|每个 AI、每个工具单独写一套|按统一标准实现一次|
|工作量|N×M，成倍增长|N\+M，一次到位|
|复用性|换个 AI 就得重写|写好的工具到处能用|
|谁来用|只有开发者能接|装好后普通用户也能用|

|**💡 小贴士**<br>一句话记住 MCP：它是“AI 应用”和“外部工具/数据”之间的通用接口标准。你把能力按这个标准做成一个“服务器”，任何支持 MCP 的 AI 都能即插即用。|
|---|

## 9\.2　工作原理：记住三个角色

MCP 的世界里只有三个角色，搞清楚它们的分工，后面写代码就一通百通：

|**角色**|**是谁**|**负责什么**|
|---|---|---|
|主机 Host|你和 AI 对话的程序，如 Claude 桌面端|接收你的提问，调度模型和工具|
|客户端 Client|主机内部的一个连接模块|负责和“服务器”建立并维持连接|
|服务器 Server|你自己写的那个程序|对外提供工具、资源、提示词|

**这里有一个新手最容易搞反的关键点：**MCP 服务器是“被动”的提供方。它自己不会去调用大模型，而是安安静静等着客户端来问“你有哪些工具？”，然后由模型来决定要不要用、用哪一个。

**换句话说，调用的方向是“模型 → 工具”，不是“工具 → 模型”。**你只管把能力写成工具暴露出去，至于什么时候用、传什么参数，全交给模型判断。下面这张图就是一次完整调用的流程：

*图 6\-1　一次工具调用的完整流程*

**你提问  →  主机把“问题＋可用工具清单”交给模型**

**→  模型决定调用某个工具并填好参数**

**→  客户端按 MCP 协议调用你写的服务器**

**→  服务器执行、返回结果  →  模型用大白话回答你**

|**💡 小贴士**<br>理解了这个方向，你就不会再纠结“我的服务器里要不要写 AI 的逻辑”——完全不用。服务器只是个老实的“工具箱”，AI 在外面负责动脑子。|
|---|

## 9\.3　MCP 能给 AI 的三样东西

一个 MCP 服务器可以对外提供三类东西。新手先重点掌握第一种“工具”，另外两种了解即可：

|**类型**|**是什么**|**举例**|
|---|---|---|
|工具 Tools|能执行动作的函数（最常用）|记一笔账、发邮件、查天气|
|资源 Resources|只读的数据，模型可随时读取|一个配置文件、一段说明文档|
|提示词 Prompts|预设好的模板指令，像“快捷指令”|“帮我生成本月账单报告”|

**本课我们只用“工具”这一种就能做出完整的记账助手。**在代码里，这三样分别用 @mcp\.tool\(\)、@mcp\.resource\(\)、@mcp\.prompt\(\) 三个“装饰器”来声明，写法高度一致，学会一个就会三个。

## 9\.5　实操：写一个“记账小助手”服务器

目标：做一个 MCP 服务器，给 AI 三件能力——记一笔账、查看账目、统计花销。我们把数据存在一个 JSON 小文件里。新建一个文件 accounting\_server\.py，跟着下面五步往里写。

**第 1 步：导入并创建服务器实例。**FastMCP 是官方包里自带的“快捷框架”，一行就能起一个服务器。

|**accounting\_server\.py（第 1 段）**<br>import json<br>from pathlib import Path<br>from datetime import datetime<br>from mcp\.server\.fastmcp import FastMCP<br><br>\# 创建一个 MCP 服务器<br>mcp = FastMCP\("记账小助手"\)<br><br>\# 账目存到脚本同目录下的 expenses\.json<br>DATA\_FILE = Path\(\_\_file\_\_\)\.parent / "expenses\.json"|
|---|

**第 2 步：写两个读写文件的小帮手。**它们不是工具，只是给后面的工具调用的内部函数。

|**accounting\_server\.py（第 2 段）**<br>def \_load\(\):<br>if DATA\_FILE\.exists\(\):<br>return json\.loads\(DATA\_FILE\.read\_text\(encoding="utf\-8"\)\)<br>return \[\]<br><br>def \_save\(records\):<br>DATA\_FILE\.write\_text\(<br>json\.dumps\(records, ensure\_ascii=False, indent=2\),<br>encoding="utf\-8"\)|
|---|

**第 3 步：写第一个工具——记账。**注意函数上面那行 @mcp\.tool\(\)，它是把普通函数“变成 AI 可用工具”的关键。

|**accounting\_server\.py（第 3 段）**<br>@mcp\.tool\(\)<br>def add\_expense\(amount: float, category: str, note: str = ""\) \-\> str:<br>"""记一笔账。amount 金额\(元\)，category 分类\(如 餐饮/交通\)，note 备注\(可选\)。"""<br>records = \_load\(\)<br>records\.append\(\{<br>"date": datetime\.now\(\)\.strftime\("%Y\-%m\-%d %H:%M"\),<br>"amount": amount,<br>"category": category,<br>"note": note,<br>\}\)<br>\_save\(records\)<br>return f"已记录：\{category\} \{amount\} 元（\{note or '无备注'\}）"|
|---|

|**💡 小贴士**<br>装饰器 @mcp\.tool\(\) 之所以神奇，是因为模型完全靠这个函数的“三件信息”来读懂你的工具：① 函数名（add\_expense）就是工具名；② 参数的类型注解（amount: float）告诉模型该传什么；③ 三引号里的说明文字（docstring）告诉模型这个工具是干嘛的。所以函数名、注解、说明都要写清楚——这等于是写给 AI 看的“说明书”。|
|---|

**第 4 步：再加两个工具——查账目、做统计。**写法和上面一模一样，每个能力一个带 @mcp\.tool\(\) 的函数。

|**accounting\_server\.py（第 4 段）**<br>@mcp\.tool\(\)<br>def list\_expenses\(category: str = ""\) \-\> str:<br>"""列出账目。传 category 只看某分类；不传则列出全部。"""<br>records = \_load\(\)<br>if category:<br>records = \[r for r in records if r\["category"\] == category\]<br>if not records:<br>return "暂无记录。"<br>return "\\n"\.join\(<br>f'\{r\["date"\]\} \| \{r\["category"\]\} \| \{r\["amount"\]\}元 \| \{r\["note"\]\}'<br>for r in records\)<br><br>@mcp\.tool\(\)<br>def summary\(\) \-\> str:<br>"""统计总支出，以及各分类的金额合计。"""<br>records = \_load\(\)<br>if not records:<br>return "暂无记录。"<br>total = sum\(r\["amount"\] for r in records\)<br>by\_cat = \{\}<br>for r in records:<br>by\_cat\[r\["category"\]\] = by\_cat\.get\(r\["category"\], 0\) \+ r\["amount"\]<br>lines = \[f"总支出：\{total\} 元", "分类明细："\]<br>for cat, amt in sorted\(by\_cat\.items\(\), key=lambda x: \-x\[1\]\):<br>lines\.append\(f"  \{cat\}：\{amt\} 元"\)<br>return "\\n"\.join\(lines\)<br><br>@mcp\.tool\(\)<br>def delete\_last\(\) \-\> str:<br>"""删除最近记的一笔账。"""<br>records = \_load\(\)<br>if not records:<br>return "暂无记录，无法删除。"<br>removed = records\.pop\(\)<br>\_save\(records\)<br>return f"已删除最近一笔：\{removed\['category'\]\} \{removed\['amount'\]\} 元（\{removed\['note'\] or '无备注'\}）"<br>|
|---|

**第 5 步：在文件最后加上启动代码。**mcp\.run\(\) 会让服务器跑起来、等待主机来连接。

|**accounting\_server\.py（第 5 段）**<br>if \_\_name\_\_ == "\_\_main\_\_":<br>mcp\.run\(\)|
|---|

**到这里，一个完整的记账 MCP 服务器就写好了——**总共不到 50 行。三个工具 add\_expense、list\_expenses、summary，就是我们交给 AI 的三件能力。

## 9\.7　回顾：一次调用到底发生了什么

把前面学的串起来，当你说“记一笔午饭 35 元，餐饮”，幕后是这样跑的：

- ① dify（主机）发现你装了一个有 add\_expense 工具的服务器；

- ② 模型判断这件事该用 add\_expense，并填好参数 amount=35、category=餐饮；

- ③ 客户端按 MCP 协议把这次调用发给你的服务器；

- ④ 服务器执行函数，把记录写进 expenses\.json，返回“已记录”；

- ⑤ 模型拿到结果，用自然语言回复你“好的，已记下午饭 35 元”。

**看懂这条链路，你就真正理解了 MCP——**AI 负责“听懂和决策”，你的服务器负责“干活”，MCP 协议在中间传话。

|**❓ 常见疑问**<br>问：MCP 服务器和前几课的 Dify 应用有什么关系？<br>答：可以理解为互补。Dify 让你快速搭出“会聊天/会出活”的应用；MCP 则专门解决“让 AI 去调用外部工具和数据”。很多平台（包括 Dify）都已支持把 MCP 服务器接进来，作为应用的“外挂能力”。|
