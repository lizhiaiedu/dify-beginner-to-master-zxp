## 12\.2　发布操作

1. 在应用编排页右上角，点击“发布 / 更新”，确认最新版本已发布。

2. 点击旁边的“运行 / 访问”，能拿到公开访问链接，复制就能分享。

3. 如需挂到网站，找到“嵌入 / Embed”，复制那段代码交给网站维护者粘贴即可。

4. 每次改动后，都要再点一次“发布 / 更新”，线上版本才会同步。

|**⚠️ 避坑提醒**<br>“改完即发布”要养成习惯。新手最常见的困惑就是“我明明改好了，别人怎么看到的还是旧的”——十有八九是忘了点发布。|
|---|

## 12\.3　API 是什么（一分钟理解）

**API** 可以理解成你的应用对外开的一个“服务窗口”。别的程序按规定格式来这个窗口“点单”（发送问题），你的应用就把 AI 的回答“出餐”（返回结果）。

比如你公司有自己的 App，想把 Dify 做的客服能力放进去，就让程序员通过 API 调用——用户在你 App 里聊天，背后其实是 Dify 在干活。新手不必会写，但要知道“能这么用”，将来和技术同事沟通时心里有数。

|**💡 小贴士**<br>区分两种 Key：模型的 API Key（第 3 课）是“你去调模型”；应用的 API 密钥（在应用“访问 API”页生成）是“别人来调你的应用”。两者是两回事，但同样都要保密。|
|---|

## 12\.4　上线前检查清单

正式分享给别人之前，过一遍这份清单，能省去很多事后救火：

- 发布了吗：最新改动是否已点“发布 / 更新”；

- 测过吗：用常见问题和刁钻问题各试几遍，看回答是否靠谱；

- 防编造了吗：挂知识库的应用，提示词里有没有“查不到就说没找到”；

- 额度够吗：接入的模型账户余额/免费额度是否充足，避免用着用着停摆；

- 范围对吗：有没有把不该公开的内部资料，放进了对外应用里。

## 12\.5　综合实战：智能简历助手（贯穿全课）

用这个项目把整门课的知识串起来。目标：用户上传招聘 JD 和自己的简历，助手给出“匹配度分析 \+ 简历优化建议”。

### 设计思路（先想清楚再动手）

- 用什么类型：多步骤、要分析，用工作流；

- 要不要知识库：建一个“简历写作技巧”知识库供参考；

- 流程怎么走：开始\(收 JD\+简历\) → LLM\(提取 JD 要求\) → 知识检索\(写作技巧\) → LLM\(对比分析给建议\) → 结束\(输出报告\)。

### 动手步骤

1. 建一个“简历写作技巧”知识库（第 7、8 课），上传几篇优质资料。

2. 创建工作流应用，开始节点设两个输入：jd（招聘要求）、resume（简历内容），都用“段落”类型。

3. 第一个 LLM 节点，提示词让它从 \{\{jd\}\} 中提炼“岗位核心要求”：

|**示例·第一个 LLM 节点提示词**<br>*请从下面的招聘要求中，提炼出该岗位最核心的 5 条硬性要求和技能关键词，分点列出：\{\{jd\}\}*|
|---|

1. 知识检索节点，连上“简历写作技巧”知识库。

2. 第二个 LLM 节点，综合岗位要求、检索到的技巧和 \{\{resume\}\}，输出分析报告：

|**示例·第二个 LLM 节点提示词**<br>*你是资深简历顾问。请对照岗位核心要求，分析这份简历【\{\{resume\}\}】的匹配度，并参考写作技巧给出改进建议。输出：① 匹配度评分（满分 10）；② 三条最关键的优化建议。分点、客观、不超过 300 字。*|
|---|

1. 结束节点，输出最终报告。运行测试，再发布。

|**💡 小贴士**<br>这个项目用到了：变量、提示词工程、知识库、多 LLM 串联、工作流编排——几乎是整门课的总复习。把它独立做出来，你就算真正入门了。|
|---|

## 12\.6　学完之后，往哪走

- 挑一个你做过的应用，按“真正好用”的标准继续打磨、发布给真实用户试用；

- 回到“探索”翻更多模板，遇到没学过的节点就点开研究；

- 关注 Dify 官方文档和社区，版本更新很快，新功能常有惊喜；

- 把课程附录的“避坑清单”和“术语表”留着，日后随时查。

|**❓ 常见疑问**<br>问：发布后要花钱吗？<br>答：应用本身免费，但每次有人使用都会消耗你接入的那个模型的额度（按 API Key 计费）。用得多就需要给模型账户充值。可以先用免费额度跑，量大了再评估成本。|
|---|

|**❓ 常见疑问**<br>问：免费额度用完了，应用会怎样？<br>答：模型会调用失败，应用就答不出来了。所以上线前要确认额度，重要应用记得提前充值或设置提醒，别让它“断粮”。|

# 附录 A　新手避坑清单（建议贴在显眼处）

这些是新手最常踩的坑，对照检查能省下大量排错时间：

|**现象 / 场景**|**原因**|**解决办法**|
|---|---|---|
|应用没法回答 / 报模型错误|没接模型或模型 Key 失效|去“模型供应商”检查模型是否“可用”|
|建知识库报错没有向量模型|没接 Embedding 模型|接一个向量模型再建库|
|改了东西别人看不到|没点“发布 / 更新”|改完一律重新发布|
|知识库答非所问|分段不合理或没召回测试|调分段、重新索引、做召回测试|
|AI 编造内容|提示词没约束|加“资料没有就说没找到，别编造”|
|工作流运行报错|节点没连线或变量名不对|检查连线和变量引用是否一致|
|回答太啰嗦/太发散|没限制 / 温度过高|提示词限字数，调低温度|
|API Key 泄露被盗刷|截图或贴到公开处|立即作废重生成，永不公开|

# 附录 B　核心术语速查表

看到不认识的词，回这里查：

|**术语**|**大白话解释**|
|---|---|
|大模型 / LLM|被海量文字训练出来的“超级大脑”，能理解和生成语言|
|提示词 / Prompt|你给 AI 的“工作说明书”，决定它怎么干活|
|API Key|调用模型的“钥匙 \+ 记账卡”，要保密|
|变量|提示词里的“填空格”，用户填什么就代入什么|
|温度 / Temperature|回答的发散程度，低=稳重，高=有创意|
|知识库 / RAG|给 AI 当参考书的资料库，让它“先查再答”|
|分段 / 切块|把长文档切成小块，方便检索|
|向量 / Embedding|把文字转成数字，方便“按意思找资料”|
|召回 / 检索|从知识库里找出和问题相关的片段|
|工作流 / Workflow|把任务拆成节点、连成流程的可视化编排|
|对话流 / Chatflow|聊天形式的工作流，每轮回复背后跑一遍流程|
|节点 / Node|工作流里的一个功能方块，各管一件事|
|Agent / 智能体|能自己决定步骤、调用工具完成目标的 AI|
|工具 / Tool|给 AI 调用的外部能力，如搜索、计算、画图|
|发布 / Publish|把应用对外提供使用的动作|

# 附录 C　学完之后怎么继续进阶

## 给自己的练习清单

- 把你工作中一件重复的事，做成一个 Dify 应用，真正用起来；

- 挑一个应用，反复打磨提示词，体会“细节决定质量”；

- 尝试给知识库换不同分段方式，建立对“检索质量”的直觉；

- 用工作流复刻一个你常用的多步骤流程。

## 值得进一步探索的方向

- **自定义工具：**把公司的接口接进来，让 AI 能查你的真实业务数据。

- **多应用协作：**把工作流封装成工具，被 Agent 调用，搭出更强的组合能力。

- **团队协作与运维：**学会看日志、分析数据、控制成本，让应用稳定长期运行。

- **关注版本更新：**Dify 更新很快（如实时协作、MCP 工具等新特性），常去官方文档看看新功能。

|**💡 小贴士**<br>最后一句话送给你：AI 应用开发，最重要的不是记住每个按钮在哪，而是养成“把一件事拆成步骤、想清楚每步要什么”的思维。工具会变，这个能力不会过时。祝你做出真正好用的 AI 应用！|
